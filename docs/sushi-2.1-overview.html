two results, cached

Popular ⇄ Custom

The first deal is Popular: dietary profile + what online reviewers actually praise. Run the refine sheet's narrowing questions and you get Custom. Both stay cached per scan — the segmented toggle switches between them instantly, for free.

⭐ Crowd favorites the swap

Cached reviews fold straight into the first ranking. Uncached? The picks render instantly anyway and one background search folds in ★ badges a beat later — no re-rank, no shuffle.

✎ Refine sheet

On-device narrowing questions (zero tokens) shrink the pool; one re-rank builds your Custom order without ever touching the cached Popular result.

🎛️ Intelligent tune chips 0 tokens/tap

$ lower · Light meal · Safe bet · Surprise me. The ranker returns a slate of up to 8 picks tagged with which tunes each suits — a chip re-deals the whole hand from the slate, instantly.

One toggle flips a contender into a deterministic "stretch pick" — a safe-but-bolder dish sharing a flavor lane with your matches, dashed in plum.

design principles

The shape of the engine

A UI-free engine package (src/engine/) does all the thinking. The model is consulted exactly twice on the golden path; everything the user touches afterwards is deterministic, on-device, and instant.

Safety is never delegatedA deterministic hard gate enforces allergens and religious rules. The model ranks only pre-cleared dishes.
Pay for judgment onceOne vision read, one rank. Tunes, stretch picks, and view switches re-use that judgment for free.
Honest by constructionLive loaders show real stages; blocked dishes are listed with reasons; every cent is in the ledger.
the pipeline · one scan
📐

Photo prep on-device

resize + compress

The shot is resized for a fast, cheap read before any tokens are spent.

📖

Vision read model call

vision.read

One multimodal call reads the entire menu into structured dishes — name, price, ingredients, flavor profile, spice level, dietary tags, a rough protein estimate — plus menu context: restaurant name, cuisine, signature items, and whole-menu notes like a halal certification.

🛡️

Hard dietary gate deterministic · 0 tokens

applyHardGate · runs ONCE per scan

Structured constraints (parsed once from the user's free-text profile) split the menu three ways: allowed, verify (uncertain — carries "ask staff" reasons), and blocked. Blocked dishes never enter any candidate pool, any ranking prompt, or any review citation — by construction.

⭐

Crowd reviews cache first 1 search if cold

reviews.fetch · fuzzy match on-device

The review cache is checked for free. A cache hit folds reviewer favorites straight into the ranking; a miss triggers exactly one background web search whose results appear later as ★ badges on the already-dealt cards. Matching review dish names to menu items is pure on-device string work.

👨‍🍳

The rank model call

reason.rank · runs inside the same loader

One call ranks the gated pool against the user's preferences, restaurant notes, and crowd favorites — returning a slate of up to 8 picks, each with a specific one-line "why", a match score, macro estimates with confidence, safety flags, and tune affinities: which of the four tune chips each pick genuinely suits, judged from ingredients and preparation.

⚡

Everything after is instant 0 tokens

tunes · bridgePick · spice trim · view toggle

Tune chips select a new top-3 from the slate (model tags first, deterministic keys as tie-break and backfill). Adventurous mode computes a stretch pick from shared flavor lanes. Popular ⇄ Custom switches between cached results. None of it costs a token or a spinner.

🍽️

On demand only lazy model calls

dish.detail · refine → one custom re-rank

Tapping "why?" fetches a deeper dish explanation (with tap-to-explain terms) once, then caches it. Finishing the refine sheet's on-device questions triggers the only other rank — the Custom order.

accounting

💰 Every call is metered

A client-side usage ledger records each API call's label, token counts, web searches, and real dollar cost — surfaced in the hidden stats screen. A typical scan: one vision read, one rank, and at most one review search; dish details and the custom re-rank only when asked for.

roadmap

Where Sushi 2.x goes next

Candidates drawn from the v2 spec's deferred items and design seams already built into the app — ordered by how close they are, not promises.

Near · finish the loop
Taste memory

Hold-to-lock already says "saved to your taste profile" — make it true: persist locked picks and post-meal feedback, and feed them into the next scan's ranking context.

Outcome check-in

A quiet "how was it?" the next time the app opens — one tap, closing the recommend → eat → learn loop.

Supabase persistence

Move the profile and taste history off-device so they survive reinstalls and sync across devices (explicitly deferred out of 2.1).

TestFlight round

Real menus, real lighting, real restaurants — validate the one-loader flow and tune-chip slate breadth on device.

Next · sharpen the engine
Budget & protein tunes

Revive the pre-redesign budget slider and protein-per-$ logic as first-class tune chips on the slate system.

Large-menu sectioning

Menus that overflow one read today ask for a re-shot; instead, read by section and merge — no user-visible failure.

Online menu lookup

Re-integrate the no-photo path: name a restaurant, pull its menu from the web, and run the same gate + rank pipeline on it.

Prompt caching

When the lookup loop returns, cache its repeated context (~5–30% savings on that flow — the one place caching pays).

Later · widen the table
Group ordering

Two profiles, one menu: picks that respect everyone's gates and find the shared-dish sweet spot.

Multi-photo scans

Stitch several shots (drinks page, specials board) into one candidate pool.

Wider review sources

Blend more review providers into crowd favorites, with the same on-device matching and gate rules.

Localization

Read non-English menus and explain dishes in the user's language — tap-to-explain is already the natural seam.

⚖️ Standing constraints carry forward: safety stays deterministic and on-device, the model is paid for judgment as few times as possible, loaders never lie, and every token lands in the ledger.
